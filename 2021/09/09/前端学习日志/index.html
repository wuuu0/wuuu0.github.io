前端学习日志

发表于 2021-09-09 更新于 2022-03-04 分类于 Front-end

该日志汇总了我收集的前端学习路线和学习资料，并简单记录一下我的前端学习过程。关于前者，明确的目的让你少做无用功，科学的步骤可以让你保持合适的节奏，有效的方法能则提高你的效率，时常铭记 “明确目的，保持节奏，提高效率”；至于后者，是因为学得再多，不记录不练习/使用总是会忘得很快，此外，也可以借此日志来督促自己的学习进度。

学习路线

前端学习路线

主要遵循Luke大佬给出的 前端学习路线 。此路线是他基于Roadmap to becoming a web developer in 2021 制作的，该路线图展示了你在学习前端开发过程中，你应该去学习的技术，并且作者也给出了他的建议，即哪些是他推荐去学的/哪些是不推荐或过时的，总结的非常详细。但过于详细的路线容易让新手陷入迷茫，不知道主路为何，不知道先学什么，后学什么，学到什么程度等等。

于是 Luke 整理了一个更适合新手入门的前端路径图。大佬 讲解学习路线的视频也非常不错，解释了如何使用这个线路图，也给出了清晰的学习节奏。

阶段一

浏览器原理

  • Luke的视频：浏览器是如何运作的？

    详细讲解了 从打开窗口输入 url 地址，到呈现出页面这个过程中，浏览器内部发生了什么事情。将浏览器的各个程序模块和初次加载流程讲的很清楚。
    不足之处在于，没有讲清楚 JS 在初次加载后的执行情景，没有讲到屏幕每帧刷新时浏览器会做什么。

    初次解析渲染完 html 后，JS 修改页面（代码修改dom和cssom），和重绘重排是怎么个对应关系？JS 每改变一处地方，就重新渲染一次？还是会累计一定修改再重新渲染？还是取决于屏幕刷新率？每次渲染都要拿完整的 DOM 结构从头开始？

    这些问题在第三点——JavaScript 权威指南中得到了解答

  • 红宝书第二章，介绍script标签

    defer和async script标签，到底何时执行（书里面说，defer的话，在html解析完成后执行，又说在DomContentLoad事件前执行，这不是矛盾吗；还是说顺序就是：html 解析完成 –> defer js解析执行 –> DomContentLoad 事件）

    还说异步加载的 script标签 最好不要（直接）修改页面结构，但其实是可以修改的，只是可能需要重新渲染（document.write 需要重新解析构造DOM，而document.body.appenChild 这种直接操作DOM的是否会好一些？）

  • JavaScript 权威指南 - 15.1章节 Web编程基础。

    较详细地讲解了 JS 在浏览器中的执行情景：两个阶段 首屏渲染阶段 异步阶段和事件驱动。（不只是站在JS的执行的角度，而是着眼于浏览器整体进程和线程的角度看JS的执行流程）

    补充：关于第二阶段，文章-对事件循环的理解 中提到，在事件循环中，每执行完微任务就会检查是否需要渲染——这就是我们之前疑问的JS执行和渲染时机的关系。

Html + CSS 知识

2021.4.1 - 2021.8.22

  • HTML CSS 菜鸟教程，简单过了一一遍

  • HTML CSS 系统化梳理 —— 字节青训营课程 + HTML+CSS笔记 CSS笔记整理

    • 关于布局的知识体系

      其中常规流中的盒子，在某种排版上下文（IFC、BFC等）中参与布局。排版上下文，可以理解成一个用于容纳元素div容器，它在特定的情况下创建，它自身整体和它内部的元素遵循某些排版规则。

    • IFC-行高、行级盒子（行内框）、行盒（行框）

      MDN line-height：line-height CSS 属性用于设置多行元素的空间量，如多行文本的间距。对于块级元素，它指定元素行盒（line boxes）的最小高度。对于非替代的 inline 元素，它用于计算行盒（line box）的高度。

      css基线与行高、行模型和对齐 一、行模型和对齐 二

      关键在于理清楚这些值之间的决定关系：容器的line-height，行内元素的字体大小 决定行级盒子的大小（可以将line-height当作锚点？line-height的定义锁定了两行的相对位置，不论两行中的字体大小有多少种）；行内元素的垂直对齐方式，决定了行内元素（行级盒子）之间的相对位置；同一行中最顶和最底的行级盒子边界决定了行盒的高度。

    学习一个语言，除了语法，还要理解它的工作原理。语言工作原理主要考虑 编译/解释时 和 运行时 （维基百科）

  • 总之：

    前端项目工程化形式 —Webpack—> 浏览器能理解的 html+css+js 传统项目组织形式。

    我们实际工作开发的时候，按需求配置好 Webpack 后，就是在工程化的模式/开发环境下进行理解和开发的。所以先要熟悉工程化的模式下我们是如何开发的（比如把index.html当作是浏览器默认加载的页面，index.js是js程序的入口文件-即js从它开始执行），然后或出于特别的需求或是考虑到性能优化，再进一步深入了解 Webpack 是如何打包的。

    打包后的东西，才是最终部署到服务器上的，用于生产环境的，可以直接被浏览器运行的东西，也就是 dist 下的各种静态资源和 bundle。

    PS：所谓工程化，我的理解就是 在开发的时候 能按照一些便于我们开发的模式、规定和流程来进行开发以及使用一些协助我们开发的工具（比如显式的模块化、使用方便的资源形式、易于理解和使用的工程目录、语法提示等）。开发完成后，我们再将这些开发环境下的所有东西 转化成 生产环境下的东西。实现前端工程化的目的简单来说就是通过流程规范、自动化工具来提升前端的开发效率、性能、质量、多人协作能力以及开发体验。

目前我对前端的认知是，用户在浏览器中输入URL，浏览器从服务器加载 HTML、CSS、JS 等资源，本地浏览器对获取的内容进行渲染显示，并响应用户对页面的操作。前端就负责整个页面的实现（所谓 HTML-结构内容 CSS-样式布局 JS-行为）+科学的代码风格和设计模式/理念，以及考虑在浏览器端和网络传输方面能做的性能优化，也可以了解服务端的开发使得我们写出对后台更友好的页面（甚至迈向全栈）

React Router

  • React Router 中文文档

  • React Router 尚硅谷 P74-P78

    理解 SPA

    1. 单页面Web应用（single page web application, SPA）
    2. 整个应用只有一个完整的页面
    3. 点击页面中的链接不会刷新页面，只会做页面的局部更新
    4. 数据都需要通过 ajax 请求获取，并在前端异步展现

    理解路由

    1. 什么是路由

      一个路由就是一个映射关系(key:value)。key为点击后浏览器地址栏切换后的路径, value可能是function或component

    2. 路由分类

    • 后端路由

      理解： value是function, 用来处理客户端提交的请求。

      注册路由： router.get(path, function(req, res))

      工作过程：当node接收到一个请求时, 根据请求路径找到匹配的路由, 调用路由中的函数来处理请求, 返回响应数据

    • 前端路由

      浏览器端路由，value是component，用于展示页面内容。

      注册路由: < Route path=”/test” component={Test}>

      工作过程：当浏览器的path变为/test时, 当前路由组件就会变为Test组件

    前端路由原理 + 基本使用

    基于浏览器的历史记录，即 BOM 的 history 对象。在页面点击按钮或链接，取消其默认事件处理，设置事件处理为调用 history API 对浏览器历史记录进行修改（栈结构，历史记录入栈出栈替换等）+ 给 history 对象添加历史记录变更事件的事件处理函数，让历史记录发生改变时做出相应的前端变更。

    在 react-router-dom 的帮助下，以上过程可封装为：将需要路由功能的组件（导航和展示）用一个 BrowserProuter组件 包裹需要路由功能的组件（通常是整个 < APP /> 组件，可创建多个 < BrowserRouter > 组件，不同路由器下有各自的路由规则）+ 创建设置路由链接（点了改变地址栏和历史记录）+ 注册路由（建立地址栏路径和组件映射关系，地址栏变更的时候渲染对应的组件）

    一般组件和路由组件的区别：语法写法不同不必说；一般组件是自然渲染，而路由组件是路由到它的时候渲染；我们区分存放它们的代码文件的位置（一般是 components 和 pages）；渲染实例化时接受的参数props内容不同，一般组件传入什么接收什么，路由组件包含 history, location, match 等。

    在非模块化的情况下，需要在 html 中指定 link-css + 给元素添加类别、id等标识；

Redux

  • Redux 官方文档

    Redux 是什么？

    它有助于首先理解这个“Redux”的东西是什么。它有什么作用？它帮助我解决什么问题？我为什么要使用它？

    Redux 是一个使用叫做“action”的事件来管理和更新应用状态的模式和工具库 它以集中式Store（centralized store）的方式对整个应用中使用的状态进行集中管理，其规则确保状态只能以可预测的方式更新。

    为什么要用 Redux？

    Redux 帮你管理“全局”状态 - 哪些应用程序的许多部分都需要的状态。

    Redux 提供的模式和工具使您更容易理解应用程序中的状态何时、何地、为什么以及如何更新，以及当这些更改发生时您的应用程序逻辑将如何表现. Redux 指导您编写可预测和可测试的代码，这有助于让您确信您的应用程序将按预期工作。

    我什么时候应该使用 Redux？

    Redux 可帮助您处理共享状态的管理，但与任何工具一样，它也有权衡。有更多的概念需要学习，还有更多的代码需要编写。它还为您的代码添加了一些额外代码，并要求您遵循某些限制。这是短期和长期生产力之间的权衡。

    Redux 在以下情况下更有用：

    1. 在应用的大量地方，都存在大量的状态

    2. 应用状态会随着时间的推移而频繁更新

    3. 更新该状态的逻辑可能很复杂

    4. 中型和大型代码量的应用，很多人协同开发

    并非所有应用程序都需要 Redux。 花一些时间思考您正在构建的应用程序类型，并决定哪些工具最能帮助解决您正在处理的问题。

    State 管理

    首先回顾一下 React 中没有 Redux 的，状态更新的情况。

    然而，当我们有多个组件需要共享和使用相同state时，可能会变得很复杂，尤其是当这些组件位于应用程序的不同部分时。有时这可以通过 “提升 state” 到父组件来解决，但这并不总是有效。（React 中的 Context 也是为了解决类似的问题，Context 和 Redux 的对比？）

    解决这个问题的一种方法是从组件中提取共享 state，并将其放入组件树之外的一个集中位置。这样，我们的组件树就变成了一个大“view”（即共享状态改变，共享之的组件的UI表现都会改变），任何组件都可以访问 state 或触发 action，无论它们在树中的哪个位置！（即，此前 View 和 State 都在某一个（父子）组件上，现在将 State 分离出来到另一个地方去了）

    通过定义和分离 state 管理中涉及的概念并强制执行维护 view 和 state 之间独立性的规则，代码变得更结构化和易于维护。

    这就是 Redux 背后的基本思想：应用中使用集中式的全局状态来管理，并明确更新状态的模式，以便让代码具有可预测性。
